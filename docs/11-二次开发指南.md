# 11-二次开发指南

## 开发前必读

- 先阅读 [`README.md`](../README.md)、[`docs/FunctionalModules.md`](FunctionalModules.md)、[`docs/DatabaseDesign.md`](DatabaseDesign.md)。
- 新功能必须符合现有模块边界，不要因为个人偏好重命名目录或改变架构。

## 新增业务功能流程

1. 确认功能属于现有模块或自然扩展。
2. 新增或调整 `Models` 实体。
3. 在 `StudentsManagerDbContext` 中增加 `DbSet` 和映射。
4. 如需默认数据，只能改 `DatabaseInitializer`。
5. 新增 Repository 方法。
6. 新增或扩展 Service 方法。
7. UI 通过 Service 调用，不直接操作 DbContext。
8. 更新 docs 文档。

## 新增下拉业务选项

- 院系、专业、班级、课程：优先来自对应业务表。
- 性别、状态、学期等字典：使用 `LookupOptions`。
- 默认字典值只允许在 `DatabaseInitializer` 维护。
- XAML 不得写死业务 `ComboBoxItem`。

## 新增学生相关表

- 必须包含 `StudentNo`。
- 页面展示使用学号，不展示内部 `StudentId`。
- 如保留 `StudentId` 兼容旧数据，也不能作为主要业务展示字段。

## 新增导入导出

- CSV 导出优先复用 `CsvExportHelper`。
- 导入时应在 Service 中实现合并规则和校验。
- 导入失败要捕获异常、写日志、提示用户。

## 新增权限

- 当前权限基于 `Users.Role`。
- 扩展前先确认是否仅需入口级权限。
- 不要在多个页面散落硬编码复杂权限逻辑；应逐步抽出权限服务。

## 新增日志

- 使用 [`AppLogger.Info/Warn/Error`](../StudentsManagerSystem/Common/AppLogger.cs#L9)。
- 关键业务增删改、导入导出、登录失败应记录。
- 不要记录明文密码。

## 测试建议

当前项目无测试项目。新增复杂业务时建议先建立测试项目，并优先测试 Service 和 Repository 行为。

## 常见二次开发任务

### 扩展学生档案子表能力

家庭信息、奖励记录、处分记录和体检信息已通过 [`ArchiveRecordEditWindow`](../StudentsManagerSystem/Views/StudentArchive/ArchiveRecordEditWindow.xaml.cs)、[`StudentService`](../StudentsManagerSystem/Services/StudentService.cs#L8) 和 [`StudentRepository`](../StudentsManagerSystem/Data/Repositories/StudentRepository.cs#L10) 接入 CRUD。继续扩展搜索、排序、分页或导入导出时，应复用现有 Service 校验和 Repository 数据访问链路。

### 细化权限

建议流程：定义角色能力 -> 数据库存储或字典化 -> MainWindow 和各页面统一检查 -> 文档更新。

### 引入 Migration

建议流程：安装 EF 工具 -> 创建初始迁移 -> 替换 `EnsureCreated` 策略 -> 保留统一初始化入口。
